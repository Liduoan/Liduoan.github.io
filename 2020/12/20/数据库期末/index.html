为建立数据库系统应用奠定基础

关系模型

其对应得术语：

关系术语 一般表格的术语
关系名 表名
关系模式 表头（表格的描述）
关系 （一张）二维表
元组 记录或行
属性 列
属性名 列名
属性值 列值
分量 一条记录中的一个列值
非规范关系 表中有表（大表中嵌有小表）

关系模式 = 关系名+ 属性集

Example:
Movies (title, year, length, genre) or
Movies (title: string, year: integer, length: integer, genre: string)**

SQL 区分三类关系

存储的关系，称为表。

这是通常要处理的一种关系，它在数据库中存储，可以对其元组进行查询和更新。

视图

通过计算来定义的关系。这种关系的数据并不真正存储在数据库中, 但是可以在需要的时候被构造出来。

临时表,

它在执行查询或更新时，由 SQL 处理程序临时构造。这些临时表会在处理结束后被删除，不会写入数据库中。

简单得举个栗子：

关系中的任意两个元组不能在Primary Key的所有属性或Unique 的属性集上完全相同.
任何违反该约束的操作企图都会被系统拒绝.
每张表有且仅有一个Primary Key, 但可以有任意多个 Unique 项.

对单个关系的查询。

SQL语句可多行书写。
SQL语句用分号(英文)结尾。

字符串模式匹配

字符串模式匹配 ： s LIKE p
其中 s 是字符属性或字符串，p 是包含 % 或下划线的字符串模式，
匹配规则：
p 中普通字符只和 s 中完全相同的字符匹配；
p 中% 可与 s 中 0 个或多个字符匹配；
p 中一个下划线可与 s 中一个任意字符匹配。
SELECT * FROM　salesman WHERE name LIKE ‘王%’;
SELECT * FROM　salesman WHERE name LIKE ‘王‘;

连接方式

自然连接

就是自然连接隐含特定的连接条件。

R1 NATURAL JOIN R2 －－自然连接。
R1 和 R2 之间应至少有一对同名属性，否则连接无效。
默认的连接条件是：R1 和 R2 的所有同名属性值对应相等。
例如：
Select * From salesman Natural Join salesorder
自然连接的结果中不包含同名属性的副本

自然连接可以使用on来增加连接条件

θ连接

R1 JOIN R2 ON cond －－θ连接。
例如：
Select * From salesman JOIN salesorder
ON salesman.empid = salesorder.empid
等价于：
Select * From salesman, salesorder
Where salesman.empid = salesorder.empid

自然连接和θ连接的区别

自然连接是MySQL自己就连接好了

而θ连接需要我们赋予连接关系式。

笛卡尔积

R1 CROSS JOIN R2 －－ 笛卡尔积。
例如：
Select * From salesman CROSS JOIN salesorder
等价于：
Select * From salesman, salesorder
注意：CROSS JOIN 不能用 ON 增加连接条件。

Mysql 中 连接 on 和 where 的区别

MySql 连接查询中 on 和 where 的区别

MySql 连接查询时，on 后面跟的是连接条件，再跟 and 关键就可以继续加过滤条件，这样看来是不是可以免用 where 增加过滤条件了呢？开始我也是这样以为的（老师是这么教的，且使用 on 的情况下不能使用 where）

当时存疑但没有深究，多次网上看帖写的连接查询用了 on，还用 where，所以了解了一下，现在做一个输出：
首先，老师教错了…（diss~）on 和 where 之间并没有冲突，随都用作设立条件，但有区别：

  1. on 后跟连接条件，这是规范。其次，如果再加筛选条件只针对关联表！没错，是关联表的筛选条件

    执行顺序：从表按照条件筛选，然后再进行连接；即先筛选再连接

  2. where 则针对连接后产生的临时表进行筛选

    执行顺序：先连接再筛选

所以，on 后跟关联表（从表）的过滤条件，where 后跟主表或临时表的筛选条件（左连接为例，主表的数据都会查询到，所以临时表中必定包含主表所有的字段，需要给主表加什么筛选条件，直接给临时表加效果相同）

注意：所有的连接条件都必须放在 on 后面，否则无论是 left 还是 right 连接都将不起作用（where 作用的是连接后的临时表，与连接已经无关）

因为 on 会首先过滤掉不符合条件的行，然后再进行其他运算，按理说 on 更快，允许的条件下，用 on 进行最大限度的筛选，后面 where 就避免重复筛选重复的数据，从而提升效率。还有一个重要特征：on 后的筛选条件筛选掉不符要求的字段后，对于主表中 null 结果的记录是没办法筛选掉的；而 where 则可以（关键就是 on 的筛选条件先筛选再连接，筛选时还不知道主表中的一些数据，自然没办法筛选）

出处：https://www.cnblogs.com/csyh/p/12969068.html

【自我总结

where操作是针对临时表进行筛选的。【也可以说where操作只针对一张表。

而on操作时针对左右表筛选后再去连接操作，也就是说on操作是对于两张表而言。

消除重复

投影运算可能产生重复的行。
投影时使用 distinct 可消除重复。
例如：计算员工号为“A0044”的销售员所销售的商品的种类和规格。

消除重复会带来较多的时间开销。
集合并 Union 运算结果中自动去掉重复行，欲保留重复行应使用 Union ALL。

聚集操作

关系代数中介绍了分组操作符γ。
SQL中通过GROUP BY子句提供了γ操作的所有功能。

​ 例：计算每种商品的销售数量和销售金额，并按商品号升序排列。
​ Salesitem (orderno, lineno, prodid, singlecost, quantity)

在含有聚合运算的 Select 子句中，只有在 Group By 子句中出现的属性才允许在 Select 子句中以非聚合的形式出现。
Group by 子句中不允许使用属性的别名。
